Lemmings is open source under the Apache 2.0 license. Contributions are welcome. This page covers the requirements for opening a pull request, the structure and purpose of the test suite, and how to run tests locally.

## Before opening a pull request

<Steps>
  <Step title="Run the test suite with race detection">
    All tests must pass before submitting a PR. Run:

    ```bash
    go test -race ./...
    ```

    The `-race` flag is required, not optional. The test suite is designed to be run under the race detector and some tests specifically verify atomic operation safety that only the race detector can catch.
  </Step>
  <Step title="Add tests for any new functions">
    Every new function must have a corresponding test that verifies its contract. Follow the patterns established in the existing test files. PRs without tests for new functions will not be merged.
  </Step>
  <Step title="Read TESTS.md">
    Read [TESTS.md](https://github.com/andreimerlescu/lemmings/blob/main/TESTS.md) before writing tests. It explains what the test suite is trying to prove, why each category of test was chosen, and what invariants each fuzz target enforces. Understanding this context will help you write tests that fit the suite rather than tests that only pass.
  </Step>
</Steps>

## Test suite overview

The test suite proves that what Lemmings measures is true and that what it reports can be trusted. This is not a standard "does the code work" assertion — it is a claim that organizations can use Lemmings reports to make production decisions. The tests are organized in three categories.

### Unit tests

Unit tests verify that every function and method does exactly what its documentation says it does, for all inputs in its defined range. They are:

- **Deterministic.** Every unit test produces the same result on every run, on every machine, in every environment. Network calls go to local `httptest` servers constructed and torn down within the test itself.
- **Named after contracts.** A test named `TestDetectWaitingRoom_MalformedPosition` identifies the broken promise when it fails, not a line number.
- **Minimal in scope.** Each test verifies exactly one contract.

### Fuzz tests

Fuzz tests verify that functions which parse externally-sourced data never panic, never produce logically impossible output, and always satisfy their invariants regardless of what bytes they receive. Lemmings applies fuzz testing only to functions that parse arbitrary bytes from live HTTP servers.

| Function | Invariants verified |
|---|---|
| `detectWaitingRoom` | Never panics. Position always >= 0. Position is 0 when not detected. |
| `sha512sum` | Never panics. Always returns exactly 128 hex characters. |
| `extractSitemapLocs` | Never panics. Never returns empty strings in the result slice. |
| `extractHTMLLinks` | Never panics. All returned links have the origin as prefix. |
| `resolveURL` | Never panics. Never returns URLs containing fragment identifiers. |
| `handleAuth` | Never panics. Always returns a valid HTTP status code. |
| `dashboardHTML` | Never panics. Always returns non-empty output. |
| `formatInt` | Never panics. Always returns a non-empty string. |
| `formatBytes` | Never panics. Always returns a string with a recognised unit suffix. |

### Benchmark tests

Benchmark tests verify that functions on the hot path of the swarm perform within the bounds required for Lemmings to scale. Run them explicitly when changing hot path functions.

| Benchmark | What a regression means |
|---|---|
| `BenchmarkSha512sum_1KB` / `_1MB` | SHA-512 runs on every page body, every visit. A regression degrades throughput across the entire swarm. |
| `BenchmarkDetectWaitingRoom_Normal` | Executes on every HTTP response body. A regression reduces the maximum sustainable visit rate. |
| `BenchmarkEventBus_Emit_Concurrent` | The EventBus is the shared nervous system of the swarm. Lock contention here stalls lemming goroutines across all terrains simultaneously. |
| `BenchmarkClientRegistry_Broadcast` | Broadcast runs once per second per connected dashboard client. A regression delays metric visibility. |
| `BenchmarkReporter_Ingest_Concurrent` | Ingest runs once per lemming death. Lock contention creates a queue of goroutines waiting to record results. |
| `BenchmarkSendLifeLog_Concurrent` | The non-blocking send is the handoff between a dying lemming and the result collector. Latency here holds a goroutine open past its deadline. |

## What the tests prove

### Metric accuracy

`TestIngest_StatusCodeBuckets` verifies that HTTP status codes land in the correct 2xx/3xx/4xx/5xx buckets. `TestReporter_Ingest_AccumulatesBytes` verifies byte totals. `TestReporter_Ingest_ConcurrentSafe` verifies under the race detector that concurrent ingestion from multiple goroutines produces accurate totals. The 2xx count in a Lemmings report is exact, not approximate.

### Percentile accuracy

`TestPercentile_P50_KnownDataset` and `TestPercentile_P99_KnownDataset` verify percentile computation against datasets where the correct answer is known analytically. The p99 latency in a Lemmings report is the 99th percentile of actual observed request durations, computed correctly using the nearest-rank method.

### No silent data loss

`TestSendLifeLog_IncrementsDroppedWhenBothFull` verifies that when both result channels are full, Lemmings increments `dropped_logs` rather than blocking or discarding silently. `TestCollectResults_DrainsPrimaryAndOverflow` verifies the collector drains the overflow channel completely before exiting. A report with `dropped_logs: 0` contains every visit from every lemming.

### Waiting room accuracy

Waiting room detection, position extraction, and duration timing are verified against real room package HTML — not against a mock. `TestRun_WaitingRoom_Detected`, `TestRun_WaitingRoom_Position`, `TestRun_WaitingRoom_DurationRecorded`, and `TestWaitInRoom_ExitsOnContextCancellation` together prove the waiting room data in a Lemmings report is accurate.

### Cookie jar isolation

`TestSpawnLemming_CookieJarIsolation` verifies that each lemming receives its own isolated cookie jar via `cookiejar.New()`. Session cookies set for one lemming never leak into another. This matches the behaviour of real users.

### Dashboard security

`TestNewDashboard_TokenHashNotRaw` verifies that only the SHA-512 hash of the token is stored in memory — not the raw token. `TestHandleAuth_RejectsWrongToken` verifies a wrong token returns 401. The constant-time comparison used in `handleAuth` and `requireAuth` prevents timing attacks.

## Running tests

### Standard run

Run all unit tests with race detection enabled. This is the command that must pass before any PR is opened.

```bash
go test -race ./...
```

### With fuzz seed corpus

Run all unit tests with race detection, then run the fuzz seed corpus for a specific fuzz function to cover known interesting inputs:

```bash
# Run all unit tests
go test -race ./...

# Run fuzz seed corpus for a specific fuzz function (one package at a time)
go test -fuzz=FuzzDetectWaitingRoom -fuzztime=10s .
go test -fuzz=FuzzSha512sum -fuzztime=10s .
```

### Extended fuzzing — specific function

Use when investigating a specific parsing function before a major release. Run in the package directory (not with `./...` — `-fuzz` targets a single package):

```bash
go test -fuzz=FuzzDetectWaitingRoom -fuzztime=60s .
go test -fuzz=FuzzExtractSitemapLocs -fuzztime=60s .
go test -fuzz=FuzzExtractHTMLLinks   -fuzztime=60s .
go test -fuzz=FuzzResolveURL         -fuzztime=60s .
```

### Benchmark run

Run before and after changes to hot path functions to verify no performance regression was introduced.

```bash
go test -bench=. -benchmem -benchtime=5s ./...
```

### Coverage report

Generates a coverage profile and opens it in the browser.

```bash
go test -coverprofile=coverage.out ./...
go tool cover -html=coverage.out
```

## New function policy

Every function added to the package must have a corresponding test that verifies its contract. This is not a style preference — it is a merge requirement.

<Warning>
Pull requests that add new functions without tests will not be merged.
</Warning>

When writing a new test, follow the naming convention used throughout the suite: `Test<FunctionName>_<ContractBeingVerified>`. For example, a test for `parseTarget` that verifies it handles an empty string should be named `TestParseTarget_EmptyString`. This naming convention makes failure messages identify the broken contract, not the line number.
